# テスト計画書: サーバーレス対戦ゲーム

## 1. 目的
本ドキュメントは、AWSサーバーレス構成によるオンライン対戦ゲームの品質を保証するためのテスト計画を定義します。本プロジェクトはAWS Free Tier（無料枠）を最大限活用しつつ、リアルタイム性を担保することを目的としています。

## 2. テスト対象
- **バックエンド**: AWS Lambda (Node.js), DynamoDB, API Gateway WebSocket
- **フロントエンド**: React アプリケーション
- **インフラストラクチャ**: AWS CDK によるプロビジョニング設定

## 3. 機能テスト (Functional Testing)
WebSocketを通じたリアルタイム通信とルーム管理機能の正当性を検証します。

### 3.1 テスト項目
- **接続管理**:
    - `$connect`: 正常にWebSocket接続が確立され、DynamoDBに接続IDが保存されること。
    - `$disconnect`: 切断時にDynamoDBから接続IDおよびルーム参加情報が正しく削除されること。
- **ルーム管理**:
    - `createRoom`: ランダムなルームIDが生成され、作成者がそのルームに参加した状態になること。
    - `joinRoom`: 既存のルームIDを指定して参加できること。存在しないIDの場合のエラー処理。
- **メッセージング**:
    - `sendMessage`: 同一ルーム内の全ユーザーに対して、メッセージが即座に配信されること。
    - 送信者本人以外にのみ届く、あるいはUI上で適切に区別されていること。
- **イベント通知**:
    - ユーザーが入室・退室した際に、ルーム内の他のユーザーに通知（`userJoined`, `userLeft`）が届くこと。

## 4. セキュリティテスト (Security Testing)
AWSのセキュリティベストプラクティスに基づき、システムの堅牢性を検証します。

### 4.1 テスト観点
- **最小権限の原則 (IAM)**:
    - Lambda関数に付与されているロールが、特定のDynamoDBテーブルへのアクセスとAPI Gatewayへの接続管理（ManageConnections）のみに制限されているか。
- **入力データの検証**:
    - 不正な形式のJSON、巨大なペイロード、スクリプト注入（XSS）を意図したメッセージに対する防御。
- **APIの保護**:
    - 意図しないルートへのアクセスや、WebSocketの同時接続数制限の確認。

## 5. パフォーマンステスト (Performance Testing)
リアルタイムゲームとして許容可能なレスポンス性能を検証します。

### 5.1 テスト観点
- **同時接続耐性**:
    - Free Tierの範囲内（Lambda同時実行数など）で、目標とする同時接続数（例: 50接続）を処理できるか。
- **メッセージ遅延 (Latency)**:
    - メッセージ送信から他クライアント受信までの時間が、ユーザー体験を損なわない範囲（目安 200ms以内）に収まっているか。
- **DynamoDBのパフォーマンス**:
    - `PAY_PER_REQUEST` モードにおいて、読み書きのスループットがボトルネックになっていないか。
- **Lambdaコールドスタート**:
    - 起動時の遅延がWebSocket接続のタイムアウトを引き起こさないか。

## 6. 回帰テスト (Regression Testing)
コード変更時の品質維持を目的とします。

### 6.1 自動テストの活用
- **バックエンド・ユニットテスト**: Lambdaロジックの単体テスト。
- **フロントエンド・テスト**: React Testing Libraryを用いたUIコンポーネントのテスト。
- **統合テスト**: `mock-backend` を使用した、フロントエンドと通信ロジックの結合確認。

## 7. テスト観点と合否判定基準
| 項目 | 観点 | 合否判定基準 |
| :--- | :--- | :--- |
| 機能 | ルーム作成・参加 | 全てのユーザーが同一ルームで同期されること |
| 信頼性 | 異常切断からの復旧 | 切断後に再接続し、再度ルームに参加可能であること |
| 性能 | メッセージ配信速度 | 1秒間に複数メッセージを送受信しても遅延が累積しないこと |
| 安全性 | 不正なルームID | 存在しないルームへの参加試行が適切に拒否されること |

## 8. スケジュール (期間)
| フェーズ | 予定期間 | 備考 |
| :--- | :--- | :--- |
| ユニットテスト準備 | 3日間 | 各コンポーネントのテストコード作成 |
| 機能検証 (手動・自動) | 5日間 | 主要機能の網羅的なテスト |
| 負荷・性能テスト | 2日間 | 複数クライアントによる同時接続テスト |
| 修正・最終確認 | 2日間 | 発見された不具合の修正と再テスト |

合計期間: 約2週間
